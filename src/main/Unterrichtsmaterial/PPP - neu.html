2) Projektaufbau – Überblick

Erklärung

Ein Programmierprojekt ist wie eine Schulmappe mit Fächern. Jedes Fach ist ein Ordner, in dem bestimmte Sachen liegen: der Code (die Bauanleitung für das Programm), Daten (z. B. Wörterlisten oder Punkte-Tabellen) und manchmal Einstellungen (Konfiguration).

Wenn alles gut sortiert ist, findest du schnell, was du brauchst – und dein Programm weiß auch, wo es nachschauen soll.

Darstellung


projekt-mathe-trainer/
├─ code/            ← Hier liegen die Anleitungen (Quellcode-Dateien)
├─ data/            ← Hier liegen Daten (z. B. Aufgaben, Ergebnisse)
├─ config/          ← Hier stehen Einstellungen (z. B. Sprache = "DE")
└─ docs/            ← Hier erklärst du dein Projekt (z. B. eine LIESMICH-Datei)

5) Daten lesen & schreiben – so sprechen Programme mit Dateien

Erklärung

Ein Programm ist wie eine Person in der Bibliothek: Es kann ein Buch lesen (Input), um zu wissen, was es tun soll, oder ein neues Kapitel schreiben (Output), um Ergebnisse zu speichern.

Darstellung – Lesen (Input)

Schritte beim Lesen:
1. Datei finden: Pfad prüfen (z. B. "data/aufgaben.json")
2. Datei öffnen: Sagen "ich will lesen"
3. Inhalt einlesen: Den Text holen
4. Umwandeln: Text → Daten (z. B. JSON „parsen“)
5. Datei schließen: „Ich bin fertig“

Darstellung – Schreiben (Output)

Schritte beim Schreiben:
1. Daten vorbereiten: z. B. eine Liste mit Punkten
2. Umwandeln: Daten → Text (z. B. CSV-Zeilen bauen)
3. Datei öffnen: Sagen "ich will schreiben"
4. Text speichern: in die Datei schreiben
5. Datei schließen: Speichern beenden
Tipp: Lege data/ für alles an, was du liest oder schreibst. So bleibt Ordnung.
Stolperfalle: Wenn der Pfad falsch ist oder die Datei fehlt, kann das Programm nicht lesen/schreiben. Prüfe Ordnernamen und Groß-/Kleinschreibung.

Was ist ein Algorithmus? – Von der Idee zum Pseudocode

Ein Algorithmus ist wie eine genaue Kochanleitung: Zutaten (=Eingaben), einzelne Schritte (=Anweisungen) und ein Ergebnis (=Ausgabe). Je klarer die Schritte, desto besser kann ein Computer sie ausführen – und wir können sie leicht in Code übersetzen.

2) Was ist ein Algorithmus?

Erklärung

Ein Algorithmus ist eine endliche, eindeutige und ausführbare Schrittfolge, die ein Problem löst: Sie verwandelt Eingaben in Ausgaben.

Eigenschaften (Faustregeln): Endlich (endet immer), eindeutig (keine Mehrdeutigkeiten), wirksam (jeder Schritt ist machbar), korrekt (liefert richtiges Ergebnis).
Qualitätsziele: Lesbarkeit, Wartbarkeit, Robustheit (auch bei „komischen“ Eingaben), Effizienz (Zeit/Platz – hier nur intuitiv).

Darstellung

BausteinFrageBeispiel
EingabeWas bekomme ich?Eine Liste mit Zahlen
AusgabeWas liefere ich?Die Summe aller Zahlen
SchrittfolgeWelche Schritte?Zähle alle Zahlen zusammen
RandfälleWas ist „speziell“?Leere Liste → Ergebnis 0

3) Problem verstehen & spezifizieren

Erklärung

Bevor du irgendetwas entwirfst, kläre das Ziel. Schreibe eine kurze Spezifikation: Eingaben, Ausgaben, Regeln, Randfälle und Beispiele.

Darstellung – Spezifikationsraster

AspektLeitfrageNotiz
EingabenWelche Daten kommen rein?z. B. Liste ganzer Zahlen
AusgabenWas soll rauskommen?z. B. eine Zahl (Summe)
RandfälleGibt es Sonderfälle?Leere Liste, sehr große Werte
BeispieleWie sieht ein Test aus?[2,3,5] → 10
AkzeptanzWoran merke ich „fertig“?Alle Beispiele bestehen
Tipp: Schreibe erst 3–5 Beispiel-Tests, dann den Algorithmus.
Fehlerquelle: Unklare Eingaben (z. B. „Zahlen“ – aber welche? Bereich? Negativwerte?).

4) Entwurf: Schritt-für-Schritt vorgehen

Erklärung

Kluge Entwürfe entstehen inkrementell: Zerlegen, skizzieren, prüfen, verbessern.
Das Grundmuster: Verstehen → Zerlegen → Entwerfen → Prüfen → Verstehen → Verbessern.

Darstellung – Vorgehen

  1. Verstehen: Schreibe die Spezifikation in 2–3 Sätzen.
  2. Zerlegen: Teile das Problem in Teilaufgaben (z. B. „durchlaufen“, „vergleichen“, „zählen“).
  3. Entwerfen: Skizziere die Schrittfolge als Pseudocode.
  4. Prüfen: Schreibtischtests mit deinen Beispielen.
  5. Verstehen: Überprüfe die Wirksamkeit und Robustheit deines Codes und verstehe mögliche weitere Schwachstellen.
  6. Verbessern: Randfälle ergänzen, Schritte vereinfachen, Namen klären.

5) Pseudocode: Regeln & Bausteine

Erklärung

Pseudocode ist sprachneutral und soll für Menschen leicht zu lesen sein. Nutze klare Namen, kurze Zeilen, Einrückungen und Kommentare. Bausteine: Sequenz, Verzweigung, Schleife, Funktion, Datenstruktur.

6) Beispiel A: Summe einer Liste

Erklärung

Spezifikation: Eingabe eine Liste ganzer Zahlen, Ausgabe ihre Summe. Randfall: Leere Liste → 0. Wir addieren nacheinander und behalten eine laufende Summe.

Invariante: Nach k Schritten ist s die Summe der ersten k Elemente.
Stolperfalle: Leere Liste vergessen → Ergebnis wäre „undefiniert“.

7) Beispiel B: Lineare Suche

Erklärung

Spezifikation: Finde die Position eines gesuchten Werts in einer Liste oder melde „nicht gefunden“. Wir gehen die Liste von vorne nach hinten durch und vergleichen.

Invariante: Vor jedem Vergleich wurde jedes Element links vom Index bereits geprüft.
Stolperfalle: Off-by-One bei der Abbruchbedingung index < länge.

8) Beispiel C: Kleinstes Element bestimmen

Erklärung

Spezifikation: Finde den kleinsten Wert in einer nicht-leeren Liste. Idee: Merke dir ein aktuelles Minimum und vergleiche jedes weitere Element damit.

Invariante: min ist stets das kleinste der bisher gesehenen Elemente.
Stolperfalle: Leere Liste nicht abgefangen → so dass Zugriff auf liste[0] fehlschlägt.

9) Beispiel D: Eingabe prüfen (Guard-Clauses)

Erklärung

Guard-Clauses sind frühe Rückgaben, die Sonderfälle sauber trennen. Das macht die Hauptlogik kürzer, eindeutiger und weniger verschachtelt.

Nutzen: Sonderfälle zuerst lösen → Hauptlogik bleibt linear lesbar.
Stolperfalle: Division durch 0, wenn Länge nicht geprüft wurde.

10) Tests, Invarianten & Checklisten

Erklärung

Schreibtischtests führen den Algorithmus mit Beispielwerten aus. Invarianten beschreiben Wahrheiten, die während einer Schleife immer gelten. Eine Checkliste verhindert häufige Fehler.

Darstellung – Mini-Checkliste

Eingaben klar? Typ, Bereich, leere Fälle, Duplikate, Sortierung.
Schritte eindeutig? Keine Mehrdeutigkeiten, klare Reihenfolge, Ende garantiert.
Randfälle getestet? Leere Daten, ein Element, sehr große/kleine Werte.
Invarianten notiert? Z. B. „s ist Summe der bisher gesehenen Elemente“.
Off-by-One geprüft? Start/Ende von Schleifen, Indexgrenzen, Länge-1.
Fehlermeldungen sinnvoll? Verständlich, frühzeitig, nicht stumm.

Schreibtischtests – systematisch Fehler finden

Schreibtischtests prüfen einen Algorithmus Schritt für Schritt mit konkreten Beispielen. Sie zeigen, ob die Variablen die richtigen Werte annehmen, ob Randfälle bedacht sind und ob das Ergebnis stimmt.

1) Was du am Ende kannst

Tracen: Du führst Pseudocode Zeile für Zeile mit echten Werten aus.
Fehler sichtbar machen: Du erkennst Fehlannahmen, z. B. falsche Initialwerte.
Gute Tests wählen: Du stellst eine kleine, aber aussagekräftige Testfallmenge zusammen.
Verbessern: Du formulierst Korrekturen und begründest sie nachvollziehbar.
Stolperfallen vermeiden: Du prüfst systematisch Randfälle und Off-by-One-Situationen.

2) Schreibtischtests: Ziel und Nutzen

Ein Schreibtischtest simuliert die Ausführung eines Algorithmus ohne Computer. Du notierst Eingaben, den aktuellen Zeilenschritt, die Werte wichtiger Variablen und die Zwischen- sowie Endergebnisse. So erkennst du Fehler früh – noch bevor du programmierst.

3) Vorgehen in sieben Schritten

  1. Spezifikation notieren: Eingaben, gewünschte Ausgabe, Randfälle.
  2. Testfälle auswählen: Typisch, Randfall leer, ein Element, mehrere, negative/positive Werte, Spezialwerte (0).
  3. Tabelle vorbereiten: Spalten z. B. „Schritt/Zeile“, „i“, „x“, „min“, „Kommentar“.
  4. Pseudocode Zeile für Zeile ausführen: Variablenwerte nach jedem Schritt eintragen.
  5. Ergebnis prüfen: Stimmt das Ergebnis mit der Spezifikation überein?
  6. Abweichungen markieren: Wo weicht das Verhalten ab? Welche Zeile ist Ursache?
  7. Korrektur formulieren: Schreibe eine präzise Änderung und begründe sie.

4) Beispiel: fehlerhafter Pseudocode „minimum“

Pseudocode

Schreibtischtest 1: nur positive Werte

Erwartung: Das Minimum von [5, 7, 3] ist 3.

Schritt/ZeileixminKommentar
Start––0Initialwert aus Zeile 3 (fehlerhaft)
Zeile 6 (i=0)0505 < 0? Nein → min bleibt 0
Zeile 6 (i=1)1707 < 0? Nein → min bleibt 0
Zeile 6 (i=2)2303 < 0? Nein → min bleibt 0
Rückgabe––0Falsches Ergebnis: erwartet 3

Schreibtischtest 2: gemischte Werte

Erwartung: Das Minimum von [-2, 4, -1] ist -2.

Schritt/ZeileixminKommentar
Start––0Initialwert 0
Zeile 6 (i=0)0-2-2-2 < 0 → min wird -2
Zeile 6 (i=1)14-24 < -2? Nein
Zeile 6 (i=2)2-1-2-1 < -2? Nein
Rückgabe––-2Hier wirkt der Fehler zufällig nicht.
Kernaussage: Der Initialwert min = 0 ist falsch. Bei rein positiven Listen führt er zu einem zu kleinen Minimum (0). Der Algorithmus ist damit nicht korrekt spezifiziert.

5) Korrektur & Begründung

Das Minimum muss mit einem realen Listenelement initialisiert werden. Eine übliche Lösung ist der erste Wert der Liste. Dann prüft die Schleife die restlichen Elemente.

Begründung: Die Invariante „min ist das kleinste der bisher gesehenen Elemente“ gilt nun von Beginn an (nach Zeile 3) und bleibt in der Schleife erhalten.

6) Checkliste & typische Fehler

Initialwerte prüfen: Sind Startwerte echte Eingabewerte oder sinnvoll abgeleitete Werte?
Grenzen prüfen: Start- und End-Indizes, „bis“ vs. „bis einschließlich“.
Randfälle: Leere Liste, ein Element, alle gleich, nur positiv/negativ, enthält 0.
Invarianten notieren: Was muss nach jedem Schleifendurchlauf wahr sein?
Spezialwerte: Division durch 0, Überläufe, Null-Referenzen.
Missverständnisse: Kommentar und Code müssen dasselbe sagen (hier „ab 1“ vs. Schleife ab 0).

2) Was sind Programmier-Paradigmen?

Ein Programmier-Paradigma ist ein Denkmuster für den Aufbau von Programmen. Es gibt vor, wie wir Probleme in Daten und Schritte zerlegen. Die drei großen Familien hier sind: imperativ (Zustand + Befehle), objektorientiert (Objekte mit Daten und Verhalten) und funktional (Berechnungen als Funktionen ohne Nebenwirkungen).

6) Programmier-Prinzipien: DRY, KISS, SRP

DRY – Don’t Repeat Yourself

Wiederhole Funktionalität nicht an mehreren Stellen. Ziehe gemeinsam Genutztes in eine Funktion, Klasse oder ein Modul.

KISS – Keep It Simple & Stupid

Bevorzuge einfache Lösungen. Komplexität nur dort, wo sie einen klaren Nutzen bringt.

SRP – Single Responsibility Principle (Einzelverantwortung)

Jede Einheit (Funktion, Klasse, Modul) sollte genau einen funktionale Verantwortung haben. Trenne Anzeige, Logik und Speicherung.

Tipp: Prüfe bei jeder Änderung: „Berührt dies das Thema oder mische ich Anliegen?“
Warnzeichen: Gleicher Code-Block taucht an mehreren Stellen auf → Verletzung von DRY.

7) OOP in Java – Grundlagen & Begriffe

Java-Wörter erklärt

  • Klasse: Bauplan für Objekte. Definiert Felder (Daten) und Methoden (Verhalten).
  • Objekt: Konkretes Exemplar einer Klasse zur Laufzeit.
  • Feld (auch „Attribut“): Variable, die zu einer Klasse/Objekt gehört.
  • Methode: Funktion, die zu einer Klasse/Objekt gehört.
  • Konstruktor: Methode zum Erzeugen/Initialisieren eines Objekts.
  • Sichtbarkeiten: public (überall), private (nur in der Klasse), protected (in Unterklassen), paket-privat (Standard – im selben Paket).
  • Interface (Schnittstelle): Vertrag mit Methodenköpfen ohne Umsetzung (Implementierung).
  • Package: Ordner/Namespace zur Strukturierung von Klassen.

Die drei OOP-Grundlagen

Kapselung: Daten sind privat; Zugriff nur über Methoden. Schützt vor unkontrollierten Änderungen.
Vererbung: Eine Klasse kann Eigenschaften/Verhalten einer anderen erweitern („ist-ein“ Beziehung).
Polymorphie: Eine gemeinsame Schnittstelle, verschiedene konkrete Umsetzungen (späte Bindung).

8) OOP in Java – wichtige Prinzipien in der Praxis

Einzelne Verantwortung (SRP) & Trennung der Anliegen (SoC)

Teile eine Anwendung z. B. in Domänenlogik, Datenzugriff und Darstellung. In Java bedeutet das unterschiedliche Klassen/Packages mit klaren Aufgaben.

Vermeidung von Wiederholungen (DRY)

Gemeinsames Verhalten in Basisklassen, Hilfsklassen oder durch Komposition (Objekt enthält ein anderes Objekt) wiederverwenden.

Offen für Erweiterung, geschlossen für Veränderung (OCP)

Bestehenden Code nicht anfassen, wenn sich die Regeln erweitern. In Java häufig durch Interfaces und neue Implementierungen.

Trennung der Schnittstelle von der Implementierung

Programmiere gegen Interfaces, nicht gegen konkrete Klassen. Das erhöht Austauschbarkeit und Testbarkeit.

Umkehr der Abhängigkeiten (DIP)

Höhere Ebenen hängen nicht von Details ab. Beide hängen von Abstraktionen (Interfaces). Details hängen von Abstraktionen ab.

Beachtung der Testbarkeit

Kleine Klassen, wenige Verantwortungen, Abhängigkeiten über Schnittstellen (Mockbar). Vermeide statische, globale Zustände.

OCP erfüllt: Neue Speicherarten? Neue Klassen, Interface bleibt gleich.
DIP erfüllt: Service hängt nur von Speicher (Abstraktion), nicht von Dateidetails.
Anti-Pattern: Service erzeugt selbst new DateiSpeicher() → harte Kopplung, schlecht testbar.

2) Von ausführbaren Bytes zur Turing-Maschine

Erklärung

Die Turing-Maschine ist ein einfaches Denkmodell für Computer. Es gibt ein langes Band mit Symbolen (z. B. 0, 1 oder leer), und einen Lesekopf, der genau ein Feld sieht. In jedem Schritt liest der Kopf ein Symbol, entscheidet sich nach einem Zustand und einer Tabelle für eine Aktion: ein Symbol schreiben, den Kopf nach links oder rechts bewegen, und in einen neuen Zustand wechseln. So entstehen schrittweise die Ergebnisse. Reale Computer sind viel komplexer, aber sie folgen dem selben Grundprinzip: zustandsabhängige Verarbeitung von Symbolen (Bits).

Darstellung – Turing-Maschine (vereinfacht)

Turing-Maschinen gibt es heute so nicht mehr. Neue Turing-Maschinen lesen nicht 1 Bit, sondern 1 Byte; um diese 8, 16, 32 oder mittlerweile sogar 64 Bit auszuwerten, führt der Lesekopf auf ein Transistorenfeld, also ein Feld mit elektronischen Schaltern. Diese schalten dann, je nach Kombination, einen anderen Verarbeitungskreislauf frei. Stell dir vor, du kannst mit 8 Schaltern immer nur 1 aus 28 Glühbirnen freischalten, je nachdem, welcher dieser Schalter auf an, bzw. aus steht.

3) Ein Byte, viele Bits – wie Transistoren Befehle auswählen

Erklärung

Ein Bit ist die kleinste Informationseinheit: 0 oder 1. Acht Bits ergeben ein Byte. In einer CPU steuern die Bits über Transistoren (elektronische Schalter) die Auswahl des aktuellen Befehls: Ein Instruktionsdecoder liest die Bit-Kombination (den Opcode) und schaltet intern die richtigen Signalwege frei, z. B. „ALU addieren“ oder „aus dem Speicher lesen“. Ein Byte mit 8 Bit steuert ein Transistorenfeld, welches 28 Signalwege freischalten kann.

Darstellung – 1 Byte → Decoder → Steuersignale

Merke: Eine andere Bit-Kombination bedeutet einen anderen Opcode. Der Decoder wählt damit andere Schaltpfade, ohne dass wir die Hardware umbauen müssen.

5) Warum nicht direkt in Bytes? Die Software-Krise

Kurz erklärt

Direkt in Maschinencode oder Assembler zu programmieren ist fehleranfällig und teuer. Mit wachsender Komplexität scheiterten in den 1960er Jahren viele Projekte an Kosten, Terminen und Qualität. Diese Software-Krise führte zu Methoden wie strukturiertem Entwurf, Modulen, Tests – und zu höheren Programmiersprachen, die von Menschen gelesen werden können und Werkzeuge für uns übersetzen.

Kurze Historie (Auswahl)

  • 1957 – FORTRAN (Backus): erste weit verbreitete Hochsprache für wissenschaftliche Berechnungen.
  • 1959 – COBOL (Hopper u. a.): Geschäfts-/Verwaltungsanwendungen, englischnahe Syntax.
  • 1958/60 – LISP & ALGOL 60: Listenverarbeitung; strukturiertes Programmieren.
  • 1968/69 – NATO-Konferenzen: Begriff „Software-Krise“, Geburt der Software-Ingenieurpraxis.
  • 1970 – Pascal (Wirth): Lehre, klare Strukturen, starke Typen.
  • 1972 – C (Ritchie): Systemprogrammierung, Unix-Ökosystem.
  • 1983 – C++ (Stroustrup): Objektorientierung + C.
  • 1995 – Java (Gosling): „Write once, run anywhere“ mit Bytecode/JVM.
  • 2000er – JIT/VMs reifen: Leistung und Portabilität im Verbund.

6) Von Java-Quellcode zu Bytecode (Compiler, Interpreter, JIT)

Überblick (Werkzeugkette)

Java trennt Sprache und Ausführung durch eine virtuelle Maschine. Der Weg von der Datei .java zur laufenden Anwendung umfasst drei Rollen:

Compiler (javac)

Der Compiler analysiert den Java-Quelltext (Syntax, Typen) und erzeugt Bytecode in .class-Dateien. Der Bytecode ist eine plattformunabhängige Befehlsfolge für die Java Virtual Machine (JVM). Der Compiler prüft u. a. Sichtbarkeiten, Typverträglichkeit, Ausnahmen und erzeugt Metadaten (z. B. Konstantenpool).

Interpreter (JVM)

Die JVM lädt Klassen (Classloader), überprüft sie (Bytecode-Verifier) und interpretiert Bytecode: Sie liest jedes Bytecode-Kommando und führt die entsprechende Operation auf dem JVM-Stack/Heap aus. Das ist portabel und sicher, aber zunächst langsamer als direkt übersetzter Maschinencode.

JIT-Compiler (Just-In-Time)

Der JIT beobachtet, welche Methoden „heiß“ sind (häufig ausgeführt). Diese Abschnitte werden zur Laufzeit in nativen Maschinencode für die aktuelle CPU übersetzt und optimiert (Inlining, konstante Faltung, Escape-Analyse, Registerallokierung). So verbindet Java Portabilität mit hoher Geschwindigkeit.

Vorteil: Einmal kompilierter Bytecode läuft auf vielen Plattformen unverändert.
Beachte: Start-Latenz durch Interpretation/JIT, dafür gute Spitzenleistung nach Aufwärmphase.

4) Anweisungen am Beispiel System.out.println()

Was ist eine Anweisung?

Eine Anweisung ist ein ausführbarer Schritt und endet in Java mit einem Semikolon ;. Die Anweisung System.out.println(...); ruft eine Methode auf, die Text in die Konsole schreibt und danach eine neue Zeile beginnt.

Wo einfügen?

Direkt innerhalb des Codeblocks der main-Methode (zwischen deren geschweiften Klammern).

2) Primitive vs. Objekte – der Kernunterschied

Primitive Variablen (z. B. int, double, boolean) enthalten direkt einen Wert. Objektvariablen (z. B. String, eigene Klassen) enthalten eine Referenz – also einen Zeiger auf ein Objekt im Heap.

Merke: Primitive kopieren Werte; Objektvariablen kopieren Referenzen.
Vergleich: a == b vergleicht bei Objekten die Referenzen (gleiches Objekt?), equals vergleicht Inhalte (wenn überschrieben).

3) Stack & Heap – wer speichert was?

Der Stack hält Aufruf-Infos von Methoden (Frames) und deren lokale Variablen (inkl. Referenzen). Der Heap enthält alle Objekte (z. B. Instanzen deiner Klassen, Arrays). Felder eines Objekts liegen ebenfalls im Heap – egal ob primitiv oder Referenzen.

Einfach merken: Lokale Variablen/Referenzen → Stack; Objekte/Arrays → Heap.
Fortgeschritten: JIT kann Objekte optimiert „auf dem Stack“ behandeln (Escape-Analyse) – das ist für Anfänger unsichtbar und ändert das Modell nicht.

4) Referenz & „Adresse“ – was zeigt wohin?

Eine Referenz ist ein Verweis auf ein Objekt. Umgangssprachlich sagt man oft „Speicheradresse“. In Java siehst du diese Adresse nicht direkt. Du arbeitest immer über die Referenz: p.name, p.methode(). Zeigt eine Referenz auf null, zeigt sie auf kein Objekt.

Null-Sicherheit: Vor p.methode() prüfen, ob p != null – sonst NullPointerException.
Nicht verwechseln: hashCode() ist nicht die Speicheradresse.

5) Lebensdauer & Garbage Collector

Lokale Variablen sterben, wenn ihr Block endet (also an der }).

Objekte auf dem Heap leben weiter, solange es mindestens eine Referenz auf sie gibt. Wenn keine Referenz mehr zeigt, gelten sie als unerreichbar und der Garbage Collector (GC) entfernt sie automatisch.

Wenn man stark vereinfacht, dann vergleicht der Garbage Collector Stack (Referenz) und Heap (Datenobjekte) und sammelt alle Datenobjekte ein, die auf dem Heap, aber nicht auf dem Stack vorkommen.

Praxis: Setze Referenzen auf null selten bewusst; meist genügt, Variablen auslaufen zu lassen.
Ressourcen: Dateien/Streams aktiv schließen – GC räumt Speicher, nicht externe Ressourcen.

6) Codebeispiele: Kopie (primitive) vs. geteiltes Objekt (Referenz)

A) Primitive – Werte werden kopiert

B) Objekte – Referenzen zeigen aufs gleiche Exemplar

Faustregel: „=“ kopiert bei Objekten die Referenz, nicht den Inhalt. Für Kopien brauchst du eine eigene Methode/Konstruktor.
Vergleichsoperator: == bei Objekten nur nutzen, wenn wirklich Referenzgleichheit gemeint ist – sonst equals.

2) Einführung: Idee & Aufbau der switch-Anweisung

switch vergleicht einen Wert (z. B. Zahl, char, String, enum) mit mehreren Fällen (case). Beim ersten passenden Fall wird dessen Block ausgeführt. In der klassischen Form beendet break den switch, andernfalls würde der nächste Fall „hineinfallen“ (Fall-through). Der default-Zweig deckt den Standardfall ab.

Stolperfalle: Ohne break laufen die folgenden Fälle weiter. Das führt oft zu doppelten Ausgaben.

5) Bubble-Sort – einfach erklärt & Pseudocode

Beim Bubble-Sort „blubbern“ große Werte Schritt für Schritt nach rechts: Man vergleicht benachbarte Elemente und vertauscht sie, wenn sie in falscher Reihenfolge sind. Nach jedem Durchlauf steht der größte Restwert am Ende. Da dieser letzte Wert dann schon an der richtigen Stelle steht, kann man die innere Schleife bei jedem Durchlauf um ein Element verringern.

Die äußere Schleife dient also dazu, dass die innere Schleife eins nach unten hezählt wird, während die innere Schleife das Array durchläuft.

2) Einführung: Math & statische Methoden

Math ist eine Utility-Klasse. Ihre Methoden sind statisch, d. h. sie gehören zur Klasse selbst und werden ohne Objekt aufgerufen: Math.sqrt(16). Statische Methoden haben Besonderheiten:

  • Kein Objekt nötig (kein new), Aufruf über den Klassennamen.
  • Kein Zugriff auf Objektzustand (this existiert nicht).
  • Häufig „reine Funktionen“ (gleicher Input → gleicher Output, keine Nebenwirkungen).
Konstanten: Math.PI, Math.E.
Winkelmaß prüfen: sin/cos/tan erwarten Radiant – nutze Math.toRadians.

2) Klasse und Objekt – die Idee

Eine Klasse ist ein Bauplan für Objekte: Sie beschreibt, welche Daten (Felder) und Fähigkeiten (Methoden) die Objekte haben. Ein Objekt ist eine konkrete Instanz dieser Klasse zur Laufzeit – mit eigenen Feldwerten, also individuell veränderbaren Daten.

3) Aufbau einer Klasse

  • Datenfelder (Attribute): Beinhalten den Zustand (z. B. String name).
  • Konstruktor(en): Initialisiert neue Objekte (z. B. ohne Parameter, mit allen Parametern, Copy-Konstruktor, siehe unten im Code zur Klasse Aufgabe). Initialisierung bedeutet zum Beispiel, dass für das Datenfeld eines Objekts Speicherplatz bereitgestellt wird.
  • Methodentypen:
    • Getter/Setter: Lesen/setzen Felder kontrolliert.
    • API-Methoden: Öffentliche Methoden der Klasse.
    • Private Hilfsmethoden: Zerlegen komplexer Aufgaben, nur intern nutzbar.

4) Verkürztes UML: Beispiel-Klasse als Diagramm

5) Sichtbarkeiten & Scope

  • private: Nur innerhalb der Klasse sichtbar (Standard für Felder!).
  • public: Von überall sichtbar (sparsam für API-Methoden einsetzen).
  • package (ohne expliziten Modifier): Innerhalb desselben Pakets sichtbar.
  • protected: Innerhalb desselben Pakets und für erbende Klassen sichtbar.
  • Scope: Der Gültigkeitsbereich einer Variable (z. B. innerhalb eines Methodenblocks oder der ganzen Klasse).

6) Konstruktor-Varianten

  • Ohne Argumente: Setzt sinnvolle Standardwerte.
  • Alle-Argumente: Initialisiert alle Felder in einem Schritt.
  • Copy-Konstruktor: Erzeugt eine Kopie aus einem vorhandenen Objekt (tiefe vs. flache Kopie beachten).
